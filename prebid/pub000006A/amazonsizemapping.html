Googletag sizeMapping NOTE: These new integration methods do not require the use of the googletag sizeMapping feature, you can implement it with a standard googletag setup for an easier header bidding integration The apstag.fetchBids() API call is compatible with your existing googletag sizeMapping configuration, and may even result in a simpler integration all together. When you define your googletag slots, you apply the sizeMapping breakpoints to the slots you wish to be responsive, then the googletag code reads the browser width and height and dynamically requests specific ad sizes from DFP for the slots that have the sizeMapping enabled. The trick with header bidding, is matching the slot sizes in the Amazon Publisher Services (APS) bid request to the ad sizes that will be ultimately requested from DFP for each slot. If this doesn’t happen, there will be a mismatch of ad sizes requested from APS and ad sizes requested from DFP. Example: If you have a sizeMapping configuration like this: var mapping = googletag.sizeMapping(). addSize([1024, 800], [[300, 600]]). // browsers with a WxH greater than 1024x800, request 300x600 from DFP addSize([800, 400], [300, 250]]). // browsers in between 800x400 and 1024x800, request 300x250 from DFP addSize([0, 0], []). // browsers smaller than 800x400, don't request an ad build(); Then for every ad slot with this sizeMapping configuration applied to, in the bid request to APS for those slots, you will need to match the sizes that will be eventually requested from DFP based on the browser screen size. It would be inefficient to just send 300×250 and 300×600 on every slot request because only one of those sizes would have the potential of actually rendering. In order to achieve this, we have developed a new feature that allows you to just pass us the googletag slot object(s) in the bid request. We will parse through the sizeMapping breakpoints, and then match the sizes in the bid request to the sizes that will be requested from DFP for each slot. There is one caveat here, the slots being requested need to already be defined so that our code can access the slot object(s) and extract the proper sizes. This means that our integration will need to be pushed into the googletag.cmd queue so that our bid request is guaranteed to execute after the slots have been defined. There are two new methods to integrate: Don’t pass us any GPT slot information in the fetchBids API call, and we will construct the bid request for all defined slots Send us an array of comma separated GPT slot objects as the slots value in the first argument to fetchBids Integration Instructions Step 1: Add new parameter to apstag.init() Step 2: Define googletag slots and apply sizeMapping Step 3: Add fetchBids() API call to request bids from APS Integrate our new code directly underneath where you define your googletag slots to allow our bid request to go out as soon as possible. If you are using another header bidding solution in conjunction with apstag, please see the other resources on how to integrate alongside other header bidders. If this is the case, then in order to leverage these new integration methods, you must either (1) push all header bidding code into the googletag.cmd queue to ensure that all the bidding executes at the same time, or (2) apply a timing mechanism to wait until we have returned our bids or a timeout has been reached before the page makes all the DFP requests.
